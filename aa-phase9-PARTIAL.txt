# Bug 1 (FIXED) — multi-consumer admin propagation
Root cause: `BoundedInMemoryPubSubTopic.consume()` short-circuited on
`target < p.lowWaterMark`, returning `Optional.empty()` even when the entry
was still physically in the queue. The integration-test path has the parent
controller self-consuming its admin topic AND the child controller remotely
consuming the same topic; the MAX-tracking lowWaterMark let the parent's
reads lock out the child for offsets the child had not yet read.

Without admin propagation, the child controller never observed ADD_VERSION
for the user store, never triggered Helix to assign partitions to the
server, and the user-store push never reached COMPLETED.

Fix (commit c5a859a2f): removed the lowWaterMark short-circuit from
consume(). The linear walk through the deque naturally returns
Optional.empty() if the entry was actually evicted; the short-circuit was
both incorrect (locks out slower peers) and redundant.

# Bug 2 (NOT FIXED) — RT topic partition-count timeout in Samza producer
Recurring failure pattern (visible in aa-phase9-iter4.log) in
`testAAReplicationCanConsumeFromAllRegions[0..6]` and
`testAAReplicationCanResolveConflicts[0..]`:
```
com.linkedin.venice.exceptions.VeniceException: Error while constructing
  VeniceWriter for store name: test-store_<id>_rt_v1
    at VeniceWriter.<init>(VeniceWriter.java:470)
    at VeniceSystemProducer.constructVeniceWriter(...)
    at VeniceSystemProducer.start(...)
    at IntegrationTestPushUtils.getSamzaProducerForStream(...)
    at ActiveActiveReplicationForHybridTest.testAAReplicationCanConsumeFromAllRegions(line 246)
Caused by: java.util.concurrent.TimeoutException
    at CompletableFuture.timedGet(CompletableFuture.java:1960)
```
Each test fails after ~68 s wall time (30 s producer init timeout + cluster
setup). The test class's own timeout is 360 s; testBatchPushWithSeparateDrainer
hits that 360 s timeout instead.

`VeniceWriter.<init>` line 451:
```java
this.numberOfPartitions =
  CompletableFuture.supplyAsync(() -> producerAdapter.getNumberOfPartitions(topicName), threadPoolExecutor)
                   .get(30, TimeUnit.SECONDS);
```
`BoundedMockInMemoryProducerAdapter.getNumberOfPartitions` synchronously calls
`broker.getPartitionCount(topic)` which throws `IllegalArgumentException` if
the topic is missing (no lock contention; no I/O). If it threw, we'd see
`ExecutionException` wrapping IAE — but we see plain TimeoutException, which
means the supplier never returns at all. Either:
  (a) RT topic name lookup goes through a path other than the topic-map
      (e.g. partitionsFor on the consumer adapter, which isn't called here);
  (b) The threadPoolExecutor in VeniceWriter is exhausted (max 2 threads,
      core threads can time out — should not be the issue given fresh writer
      per call); or
  (c) Something else inside getNumberOfPartitions blocks (worth attaching a
      jstack at the moment of failure).

Suggested hypothesis to verify next: the RT topic
`test-store_<id>_rt_v1` is being created by the *parent* controller into the
*parent* broker only, but the Samza producer for the child region is dialing
the *child* broker, where the RT topic does not yet exist. The unbounded mock
broker would have the same `IllegalArgumentException` if asked for a missing
topic, so the failure mode might be a deeper wiring difference (e.g. the
SystemProducer's broker-address resolution).
