    Fix 6b — commit d1ccdd010: per-consumer-read deep-copy of the entire

  both produce and poll. Latency overhead is one byte[] alloc + arraycopy per
  produce and per consumer-read.

What remains to block Stage B (NOT Bug 6, two pre-existing issues):

  Block (i): Stage B's setUp.sendEmptyPushAndWait sometimes does not complete
    in dc-0 within the 5-minute timeout. The dc-0 venice_system_store_meta_store
    SIT reports STARTED → END_OF_PUSH_RECEIVED → TOPIC_SWITCH_RECEIVED but
    never `Reported COMPLETED`. dc-0 logs `[Heartbeat lag] ... lagging.
    Lag: [9223372036854775807]`. Looks like a meta-store-rt heartbeat
    propagation issue in the in-memory broker (cross-DC heartbeat consumption
    not catching up). This is the same shape as the `setUp:263` failures iter8
    sometimes saw. Run-to-run variance: bench-2 (5-min run, fresh JVM) DID
    finish setUp; bench-1 (full warmup+measurement run, fresh JVM) timed out;
    bench-4 (re-run after consumer fix) timed out again. So the empty-push
    block is intermittent.

  Block (ii): When the bench DOES progress past setUp (run 2), the AA leader
    crashes after ~533 RT messages with `Start position of 'originalBuffer'
    ByteBuffer shouldn't be less than 4` at
    ActiveActiveStoreIngestionTask:1198 (the `prependIntHeaderToByteBuffer`
    call inside producePutOrDeleteToKafka). updatedValueBytes here is
    `mergeConflictResult.getNewValue()` which can be the OLD buffer when
    `MergeUtils.compareAndReturn` picks oldValue based on hashCode comparison.
    The merge resolver always sets `resultReusesInput=true` regardless of who
    wins, so when old wins, the production path expects position >= 4 but the
    old buffer (loaded via `RawBytesChunkingAdapter.getWithSchemaId` →
    `ByteBuffer.wrap(bytes, SCHEMA_HEADER_LENGTH, ...)`, which DOES set
    position=4 — but then is read/sliced upstream and ends up with
    position=0 by the time it reaches line 1198 in some path I have not
    fully traced). This is hash-of-payload dependent and would manifest
    on Apache Kafka too.

  Both blocks are pre-existing — Phase 8's Stage B [E2E] = 0 for the same
  reasons (Bug 6 fired first and masked them). Fixing block (i) likely
  requires more understanding of meta-store-rt heartbeat propagation; fixing
  block (ii) requires a small AA-merge-resolver patch (out of Phase 9 scope).
