Pre-existing Bug 6 (NEW SHAPE for iter9): During warmup iteration 1, the leader
in dc-1 hits "VeniceException: Start position of 'putValue' ByteBuffer
shouldn't be less than 4" at
ActiveActiveStoreIngestionTask.lambda$getProduceToTopicFunction$17(:1702),
called from BoundedMockInMemoryProducerAdapter.sendMessage's synchronous
callback invocation. The leader reuses input -- so when the producer callback
runs `prependIntHeaderToByteBuffer(updatedValueBytes, getIntHeaderFromByteBuffer
(updatedValueBytes), true)`, the buffer's position is already < 4 (the int
header was already truncated by a previous iteration's callback). This halts
the SIT for the rt_v1-0 partition; canary record never becomes visible; warmup
hangs; JMH force-exits.

Verification this is pre-existing:
  - aa-phase8-alloc-inmemory.log (Phase 8 in-memory bench attempt) also has
    0 [E2E] lines and the same D2/ZK service-discovery errors.
  - Phase 8's progress notes (aa-phase8-progress.md) end at "v6 launched" with
    no confirmed [E2E] result; they did NOT validate Stage B in in-memory mode.
  - The "Start position" error is in ActiveActiveStoreIngestionTask (production
    code), NOT in the bounded broker code path; the bounded
    consumer/producer adapters faithfully store and forward the
    KafkaMessageEnvelope, including the Put.putValue ByteBuffer reference.

Hypothesis: The bounded mock producer adapter invokes the callback
SYNCHRONOUSLY on the producer thread at the moment of sendMessage, whereas
real Kafka invokes callbacks ASYNCHRONOUSLY after the network round-trip. The
synchronous invocation lets the callback's "restore the original header" logic
run BEFORE the buffer is fully consumed by downstream consumers, so a
subsequent leader re-read sees the un-padded buffer. A real Kafka producer
delays the callback until AFTER the message is durable in the broker; by then
the consumer has already deserialized into a fresh buffer. Possible iter9
fixes: (a) defer the bounded producer adapter's callback to a separate
executor (mimicking real Kafka), (b) store a deep-copy of Put.putValue inside
InMemoryPubSubMessage so the consumer sees a stable buffer, or (c) make the
"restore" callback idempotent (only truncate if currently padded).
