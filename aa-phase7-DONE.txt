## Named conclusion: (b) no client/broker knob lifts the cap

NEW finding refining Phase 6: the broker is HIGHLY IDLE
(RequestHandlerAvgIdlePercent ~87% spare capacity) at the cap. So the cap
is NOT broker CPU saturation as the Phase 6 client-side signals suggested;
it's some other in-process JVM pipeline characteristic that no individual
Kafka client/broker config knob can unblock.

The 1.77s producer accumulator queue time observed in Phase 6 was a
linger.ms=1000 artifact, NOT broker back-pressure. Exp.A confirmed this:
linger.ms=5 collapsed accumulator queue to 23ms while leaving E2E flat
(actually slightly worse).

## Production-relevant interpretation

The 142k cap on this in-process benchmark is intrinsic to running broker +
controllers + ZK + leader + drainer + parallel pool + 32GB JVM all in one
JVM process competing for CPU cores. NO Kafka tuning fixes this — proven
empirically across 5 experiments.

In real production with separate broker hosts, this artifact disappears.
The Phase 5 findings about leader-side fixes (RMD cache, newPutSupersedesAll
refactor) remain the high-priority production action.
